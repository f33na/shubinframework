# Соглашение о git-коммитах для vault

> Версия стандарта: 1.0.0

Это соглашение применяется к **vault-репозиторию** продукта — markdown-репозиторию, в котором хранятся решения (ADR), паттерны, архитектура, глоссарий, журнал и knowledge wiki. Для код-репозиториев того же продукта используется базовое соглашение о git-коммитах.

Vault — это документация, не код. У него нет сборки, тестов, рантайма, релизов. Стандартные теги вроде FEATURE / FIX / HOTFIX / MAJOR подразумевают работающий артефакт, который ломается или улучшается, — здесь они не применимы. Глаголы в vault другие: **решить**, **зафиксировать**, **уточнить**, **подгрузить**, **заменить**.

---

## Формат коммита

```
[TASK-ID/TAG] Краткое описание изменения
[TAG] Краткое описание изменения
```

- **TASK-ID** — необязательный. Используется, когда в vault-репозитории существует Issue типа vault-task (например, `VAULT-12` из шаблона `vault-task.md`). У большинства коммитов в vault его нет.
- **TAG** — обязательный. См. таблицу тегов ниже.
- **Описание** — до 72 символов, в повелительном наклонении, без точки в конце, на языке команды (по умолчанию английский).

Если нужно больше контекста, добавь тело коммита после пустой строки. Особенно полезно для ADR-коммитов — описать, что именно решает ADR.

---

## Теги

| Тег          | Что покрывает                                                        | Где                  |
| ------------ | -------------------------------------------------------------------- | -------------------- |
| `ADR`        | Добавлен новый ADR                                                   | `decisions/`         |
| `SUPERSEDE`  | Новый ADR заменяет существующий (старый помечается Superseded)       | `decisions/`         |
| `ARCH`       | Изменения в ARCHITECTURE.md (границы, инварианты, структура)         | корень               |
| `STACK`      | Изменения в STACK.md                                                 | корень               |
| `GLOSSARY`   | Изменения в GLOSSARY.md (добавления, уточнения, удаления)            | корень               |
| `TEAM`       | Изменения в TEAM.md                                                  | корень               |
| `PATTERN`    | Новый паттерн или обновление существующего                           | `patterns/`          |
| `JOURNAL`    | Запись в журнал (feedback, incident, retro)                          | `journal/`           |
| `RAW`        | Внешний источник добавлен в knowledge raw                            | `knowledge/raw/`     |
| `WIKI`       | Ручная правка в knowledge wiki (коррекция от человека)               | `knowledge/wiki/`    |
| `WIKI-AGENT` | Автоматическая компиляция wiki (AI-сгенерированный коммит, через PR) | `knowledge/wiki/`    |
| `GUIDE`      | Длинный гайд добавлен или обновлён                                   | `guides/`            |
| `AGENTS`     | Изменения в AGENTS.md или knowledge/AGENTS.md                        | корень, `knowledge/` |
| `DOCS`       | README, CONTRIBUTING, прочая мета-документация                       | разное               |
| `STYLE`      | Форматирование, опечатки, починка ссылок (без смысловых изменений)   | разное               |
| `I18N`       | Изменения в переводах                                                | `i18n/`              |
| `CHORE`      | .gitignore, gitattributes, CI-конфиги, зависимости                   | разное               |

---

## Правила выбора тега

**Несколько файлов в одном коммите — выбираем доминирующий тег.** Коммит, который добавляет ADR и обновляет GLOSSARY новым термином из этого ADR, помечается `[ADR]`, а не `[ADR/GLOSSARY]`. Обновление глоссария — вспомогательная работа; объясни её в теле коммита.

**Не объединяй несвязанные изменения.** Если хочется написать `[GLOSSARY/JOURNAL]` — это два разных коммита.

**В описании ADR-коммитов указывай номер ADR:**

```
[ADR] ADR-0024 Выбор библиотеки OAuth
[SUPERSEDE] ADR-0031 заменяет ADR-0024 (стратегия ротации изменилась)
```

**`WIKI-AGENT` зарезервирован для агента-сопровождающего wiki.** Люди никогда не используют этот тег вручную. Агент коммитит в ветку и открывает PR; ревью и мердж — действия человека. Ручные правки wiki делаются с тегом `[WIKI]`:

```
[WIKI-AGENT] еженедельная компиляция 2026-04-29

Обновлены домены: auth, billing, onboarding.
Подгружены: 4 новых raw-источника, 2 ADR, 7 закрытых Issues.
```

```
[WIKI] Перенести записи о session management из auth.md в infra.md
```

**Записи в журнал append-only по соглашению.** Редактирование существующей записи — редкий случай, обычно только опечатки сразу после создания (`[STYLE]`). Если необходимо изменить старую запись по сути — объясни почему в теле коммита.

## Ветки

- **Без ветки.** Прямой коммит в main допустим для мелких изменений документации — опечатка, один глоссарный термин, одна запись в журнал. Vault — это не production-код; ставить на каждое мелкое изменение PR-гейт добавляет трения без особой пользы. Решите с командой, если предпочитаете always-PR.
- **Feature-ветка с TASK-ID.** Когда есть Issue типа vault-task, имя ветки совпадает: `VAULT-12`. Коммиты используют полный формат: `[VAULT-12/GUIDE] Добавить гайд по траблшутингу аутентификации`.
- **Feature-ветка без TASK-ID.** Для ADR обычно не нужен tracking Issue (PR — это и есть обсуждение). Используй описательное имя ветки: `adr-0024-oauth-library`. Коммиты в коротком формате: `[ADR] ADR-0024 Выбор библиотеки OAuth`.
- **Ветки агента wiki.** Агент коммитит в `wiki/auto-ingest-YYYY-MM-DD` и открывает PR с заголовком `wiki: еженедельная компиляция YYYY-MM-DD`.

## Жизненный цикл ADR в коммитах

Типичная история ADR в git выглядит так:

```
[ADR] ADR-0024 Выбор библиотеки OAuth         (изначальный PR)
[STYLE] Исправить опечатку в ADR-0024          (правка после мерджа, допустима)
[SUPERSEDE] ADR-0031 заменяет ADR-0024         (намного позже, в PR-е ADR-0031)
```

После мерджа тело ADR изменяется только двумя способами:

1. **Изменение статуса** с "Accepted" на "Superseded by ADR-NNNN" — делается в SUPERSEDE-коммите, в том же PR, который добавляет новый ADR.
2. **Чистая правка опечаток** — тег `[STYLE]`, без смысловых изменений.

Всё остальное означает, что нужно писать новый ADR.

---

## Без bump-ов версий

Vault не версионируется. Нет тегов MAJOR/MINOR/PATCH, нет RELEASE-коммитов, нет git-тегов. Vault — это живой набор документов, а не релизный артефакт. Ветки и коммиты накапливаются; прошлые состояния восстанавливаются через git history при необходимости.

Если структуре vault нужна крупная реорганизация (редкий случай — лучше постепенные изменения через ADR), это делается одним или несколькими `[ARCH]`-коммитами в одном PR, с явным ADR, объясняющим причину.

## Примеры

```bash
# Работа с ADR
[ADR] ADR-0024 Выбор библиотеки OAuth
[ADR] ADR-0025 Место хранения токена
[SUPERSEDE] ADR-0031 заменяет ADR-0024 (стратегия ротации изменилась)

# Архитектура
[ARCH] Зафиксировать границу модуля notifications
[ARCH] Добавить инвариант: фронтенд не имеет прямого доступа к БД

# Стек и глоссарий
[STACK] Заменить Heroku на Fly.io для staging
[GLOSSARY] Добавить термин "idempotency key"
[GLOSSARY] Уточнить определение "settlement window"
[TEAM] Обновить ротацию on-call на Q2

# Журнал
[JOURNAL] feedback: интервью с клиентом X по онбордингу
[JOURNAL] incident: post-mortem утечки токенов 2026-04-12
[JOURNAL] retro: 2026-Q1

# Паттерны
[PATTERN] Добавить паттерн обработки ошибок для HTTP retry
[PATTERN] Обновить паттерн валидации форм новыми примерами

# Модуль knowledge
[RAW] Статья о best practices Stripe Connect
[RAW] Транскрипт интервью с клиентом 2026-04-22
[WIKI-AGENT] еженедельная компиляция 2026-04-29
[WIKI] Перенести записи о session management из auth.md в infra.md

# Прочее
[GUIDE] Добавить гайд по траблшутингу аутентификации
[AGENTS] Уточнить правило неизменяемости папки raw
[DOCS] Обновить README с новым списком модулей
[STYLE] Починить битые ссылки в ADR-0017
[I18N] Синхронизировать ru-перевод глоссария
[CHORE] Обновить .gitattributes для распознавания markdown в linguist

# С TASK-ID (есть Issue типа vault-task)
[VAULT-12/GUIDE] Добавить гайд по траблшутингу аутентификации
[VAULT-15/I18N] Перевести ARCHITECTURE.md на ru
[VAULT-21/PATTERN] Задокументировать новый паттерн кеширования
```

## Шпаргалка

```
Есть ветка / задача?
  ДА  → [TASK-ID/TAG] Описание
  НЕТ → [TAG] Описание

Самые частые теги по областям:
  decisions/        → ADR, SUPERSEDE
  ARCHITECTURE.md   → ARCH
  STACK.md          → STACK
  GLOSSARY.md       → GLOSSARY
  TEAM.md           → TEAM
  patterns/         → PATTERN
  journal/          → JOURNAL
  knowledge/raw/    → RAW
  knowledge/wiki/   → WIKI (человек), WIKI-AGENT (AI-компиляция)
  guides/           → GUIDE
  файлы AGENTS      → AGENTS
  i18n/             → I18N
  README, CONTRIB.. → DOCS
  только формат     → STYLE
  конфиги           → CHORE

Vault не версионируется. Никаких MAJOR/MINOR/PATCH. Никаких RELEASE.
```
